## What changed this session

### 1. FOC made optional — street_date is the anchor (decision D4, resolved)
comicreleases.com publishes per-product **on-sale** dates, not Final Order
Cutoff dates. The original validation required a FOC date and rejected every DC
row (0/283). Validation now requires `street_date` and treats `foc_date` as
optional (a null FOC is accepted; a malformed non-null value is still rejected).
- Commit `9a0968a`. DC went 0/283 → 277/283.

### 2. Landing page re-centered on on-sale date
The homepage was "This Week's FOC" (keyed on `foc_date`, next 7 days). With FOC
mostly absent it showed nothing, so `/` is now **"Upcoming Releases"** — the
soonest on-sale (street) dates, forward-looking, with a "browse by week"
overflow link. Per-row FOC badges are suppressed where FOC is unavailable
(no wall of "FOC TBD").
- Commit `3a8f12d`. New `getUpcomingReleases()` in `lib/catalog.ts`;
  `showRowFoc` flag on `DateSection`; nav/title/pull-list copy updated.
- `getFocThisWeek()` is retained for when a FOC-publishing source is added.

### 3. Generic, registry-driven ingest (decision P1, resolved toward generic)
The DC-only script became a publisher-agnostic core plus a single entry point
that covers **every** publisher in `scripts/lib/sources.ts`:
- `scripts/lib/ingest-core.ts` — the whole pipeline (fetch → extract → validate
  → upsert), parametrized by publisher.
- `scripts/ingest.ts <slug> [url]` — generic runner; pulls the current URL from
  the registry when no URL is given. `--list` shows known slugs.
- `scripts/ingest-dc.ts` — kept as a thin backward-compatible wrapper.
- Commit `3a8f12d`.

### 4. Marvel / PRH shared-header date format (the hard one)
Marvel (and other PRH publishers) don't print a date next to each product.
A **section header** states the dates for every title beneath it, until the next
header:

```
FOC 08/31/26, ON-SALE 10/14/26
WOLVERINE: PARADISE #1
FOC 07/20/26, ON-SALE 09/02/26
DAREDEVIL #6
```

Two fixes: (a) the extraction prompt now propagates a header's FOC/on-sale dates
to the products that follow it, and (b) `CHUNK_CHARS` was raised to 100k so a
typical page is a **single** extraction call — a product and its header stay in
the same context instead of being split across chunks.
- Commit `5fab023`. Marvel went 6/302 → 247/283, and this is the source of all
  254 FOC dates in the DB. DC/Image unaffected (now single-pass; DC even ticked
  up to 282/283).

## Follow-ups / open items

1. **Ingest accumulates, never prunes.** Extraction is non-deterministic and the
   idempotent upsert only dedups on identical
   `(publisher, series, issue, variant)` keys, so re-running a publisher grows
   the row set rather than replacing it (DC 303 in DB vs 282 last run). Consider
   a reconcile step: within an `ingest_runs` scope, mark/delete rows for that
   publisher+source not seen in the latest run. Until then, avoid re-running a
   publisher repeatedly.
2. **Titan (Aug) & Oni (Jul) are on older months.** Those were the latest URLs
   verified in the registry; Titan's Sept was unconfirmed and Oni's Aug/Sept
   didn't surface on comicreleases after its 2026-08-01 PRH switch (see the Oni
   note in `sources.ts` — Bleeding Cool / Comics Beat fallbacks). Pass an
   explicit URL to ingest their September data.
3. **FOC-only items are currently rejected.** Validation requires `street_date`.
   A few PRH items list only a FOC date; if we want them, relax to "street OR
   foc". Note this interacts with the homepage, which sorts by `street_date` —
   FOC-only items wouldn't appear on `/`.
4. **Pull list still groups by FOC date.** `/pull-list` groups under "No FOC
   date" for most items now. If desired, re-center it on street date like `/`.
5. **comicreleases 403 risk: not observed.** Every fetch this session succeeded
   with the default User-Agent.
